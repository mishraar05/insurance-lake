# Ops Q&A Chatbot Specification

## 1. Purpose

Provide a **natural language conversational interface** for operational users (data engineers, analysts, ops teams) to query the Insurance Lake metadata, troubleshoot failures, understand lineage, and get real-time insights without writing SQL.

**Key Goals:**
* Enable non-SQL users to query operational metadata
* Reduce time to troubleshoot pipeline failures
* Improve discoverability of feeds, jobs, and data quality rules
* Provide self-service operational intelligence
* Reduce load on engineering teams for common operational questions

---

## 2. Requirements

### 2.1 Functional Requirements

**FR-1: Operational Queries**
* Answer questions about feeds, jobs, pipelines, tables
* Query metadata tables: Feed, Job, FeedJob, DQRule, ReconRule, AuditLog
* Provide status, history, and configuration information

**FR-2: Troubleshooting**
* Identify recent failures and errors
* Show error messages and root causes
* Suggest remediation steps based on error patterns

**FR-3: Lineage & Dependencies**
* Show upstream dependencies (what feeds this table?)
* Show downstream impact (what depends on this table?)
* Explain feed-to-job-to-table relationships

**FR-4: Data Quality Insights**
* List DQ rules for a feed or table
* Show DQ check results (pass/fail/warn)
* Identify data quality trends

**FR-5: Unity Catalog Integration**
* Query UC tables, schemas, catalogs
* Show table schemas and column metadata
* Explain governance policies

**FR-6: Natural Language Understanding**
* Parse questions in natural language (not SQL)
* Handle synonyms ("pipeline" = "job" = "feed")
* Support follow-up questions (context awareness)

### 2.2 Non-Functional Requirements

**NFR-1: Response Time**
* Respond to queries in <5 seconds (95th percentile)
* Cache frequently asked questions

**NFR-2: Accuracy**
* 90%+ accuracy on metadata queries
* Clear indication when answer is uncertain

**NFR-3: Security**
* Respect Unity Catalog access controls
* Users only see tables/feeds they have access to
* No credential exposure in responses

**NFR-4: Scalability**
* Support concurrent users (10-100)
* Scale with Genie Space backend (SQL warehouse)

**NFR-5: Auditability**
* Log all queries and responses (ABC Audit)
* Track usage patterns and popular questions

## 3. Architecture

### 3.1 Deployment: Databricks Genie Space

**Technology:** Databricks Genie Space (Unity Catalog-backed conversational AI)

**Components:**
1. **Genie Space** — conversational interface
2. **Metadata Tables** — Feed, Job, DQRule, ReconRule, AuditLog (from ABC framework)
3. **Unity Catalog** — table/schema/column metadata
4. **SQL Warehouse** — executes queries generated by Genie
5. **System Instructions** — custom instructions for domain-specific understanding

### 3.2 Data Sources

**Primary Metadata Tables:**
* `{catalog}.{schema}.Feed` — feed metadata (source, format, path, schedule)
* `{catalog}.{schema}.Job` — job metadata (pipeline, status, runtime)
* `{catalog}.{schema}.FeedJob` — feed-to-job mappings
* `{catalog}.{schema}.DQRule` — data quality rules
* `{catalog}.{schema}.ReconRule` — reconciliation rules
* `{catalog}.{schema}.AuditLog` — operation history (ABC Audit)
* `{catalog}.{schema}.BalanceLog` — balance check results (ABC Balance)

**Unity Catalog Tables:**
* `system.information_schema.tables` — UC table metadata
* `system.information_schema.columns` — UC column metadata
* `system.access.audit` — UC access logs

### 3.3 Genie Space Configuration

**Space Name:** `InsureLake Ops Q&A`

**Description:** 
> Ask me anything about Insurance Lake operations: feeds, jobs, pipelines, data quality, failures, lineage, and more.

**Connected Tables:**
* All metadata tables listed in §3.2
* Auto-discovery of Unity Catalog tables (user's access level)

**System Instructions:**
```
You are an expert operational assistant for the Insurance Lake on Databricks.

**Your role:**
- Answer questions about feeds, jobs, pipelines, data quality, and operational metadata
- Help users troubleshoot failures and understand errors
- Explain lineage and dependencies
- Provide insights into data quality trends

**Data sources you have access to:**
- Feed metadata (source systems, formats, schedules)
- Job execution history (runs, status, duration)
- Data quality rules and results
- Reconciliation rules
- Audit logs (ABC framework)
- Unity Catalog tables and schemas

**Response guidelines:**
- Use plain language, avoid jargon unless the user is technical
- Show SQL queries you generate so users can learn
- When showing errors, include timestamp, feed, job, and error message
- For lineage questions, explain the full chain (feed → job → table)
- Suggest next steps for troubleshooting
- If uncertain, say so and suggest manual investigation

**Example question types you can answer:**
- "What feeds failed today?"
- "Show me all DQ rules for feed X"
- "What tables depend on feed Y?"
- "Why did job Z fail last night?"
- "What's the SLA for feed A?"
- "Show me data quality trends for table B"
```

## 4. Capabilities & Use Cases

### 4.1 Feed Management

**Use Case:** Discover and understand feeds

**Sample Questions:**
* "List all active feeds"
* "Show me feeds from Salesforce"
* "What's the schedule for feed claims_daily?"
* "When was feed policies last updated?"
* "What format is feed customer_master?"

**Expected Answers:**
* Table of feeds with name, source, format, schedule, last_run
* Single feed details (FQN, path, status, SLA)

### 4.2 Job Monitoring

**Use Case:** Monitor and troubleshoot job executions

**Sample Questions:**
* "What jobs failed today?"
* "Show me the last 10 runs of job ingest_claims"
* "How long does job harmonize_policy take to run?"
* "What's the success rate for job dq_check_customer?"

**Expected Answers:**
* List of failed jobs with error messages
* Job run history (status, duration, timestamp)
* Average runtime and success rate

### 4.3 Data Quality

**Use Case:** Understand and troubleshoot DQ issues

**Sample Questions:**
* "What DQ rules are failing?"
* "Show me all DQ rules for table silver.policy"
* "What's the DQ pass rate for feed claims_daily?"
* "Why did DQ rule not_null_policy_id fail?"

**Expected Answers:**
* List of failing DQ rules with table, column, rule type
* All DQ rules for a table (rule name, type, threshold, status)
* DQ pass rate over time (trend)

### 4.4 Lineage & Dependencies

**Use Case:** Understand data flow and impact

**Sample Questions:**
* "What feeds this table?"
* "What tables depend on feed policies?"
* "Show me the lineage for gold.policy_summary"
* "If I change silver.claim, what breaks?"

**Expected Answers:**
* Upstream: feed → job → bronze → silver → gold
* Downstream: all jobs/tables that read from specified table
* Full lineage diagram (feed → pipeline chain)

### 4.5 Troubleshooting

**Use Case:** Root cause analysis for failures

**Sample Questions:**
* "Why did job ingest_claims fail?"
* "Show me errors for feed policies today"
* "What's the most common failure for job harmonize?"
* "When did feed customer_master start failing?"

**Expected Answers:**
* Error message, timestamp, job, feed
* Root cause (schema mismatch, missing file, DQ failure)
* Suggested fix (check source, validate schema, review DQ rule)

### 4.6 Unity Catalog Queries

**Use Case:** Discover and understand tables

**Sample Questions:**
* "What tables are in schema bronze?"
* "Show me the schema for table silver.policy"
* "Who owns table gold.claim_summary?"
* "What columns in table customer contain PII?"

**Expected Answers:**
* List of tables in schema
* Table schema (column names, types, comments)
* Owner, created date, storage location
* PII columns (if tagged in UC)

**Step 3: Add System Instructions**

**Step 4: Test Sample Questions**
* Test questions from §4 to validate responses
* Refine system instructions based on results

### 5.2 Query Translation Patterns

**Pattern 1: Feed Status Query**
```
User: "What feeds failed today?"

Generated SQL:
SELECT 
  feed_name,
  job_name,
  status,
  error_message,
  run_timestamp
FROM {catalog}.{schema}.Job
WHERE DATE(run_timestamp) = CURRENT_DATE()
  AND status = 'FAILED'
  AND feed_name IS NOT NULL
ORDER BY run_timestamp DESC;
```

**Pattern 2: DQ Rule Query**
```
User: "Show me all DQ rules for table silver.policy"

Generated SQL:
SELECT 
  rule_name,
  rule_type,
  column_name,
  threshold,
  status,
  last_check_timestamp
FROM {catalog}.{schema}.DQRule
WHERE target_table = 'silver.policy'
ORDER BY rule_name;
```

**Pattern 3: Lineage Query**
```
User: "What feeds table gold.policy_summary?"

Generated SQL:
WITH upstream AS (
  SELECT DISTINCT
    f.feed_name,
    f.source_system,
    j.job_name,
    j.target_table
  FROM {catalog}.{schema}.Feed f
  JOIN {catalog}.{schema}.FeedJob fj ON f.feed_id = fj.feed_id
  JOIN {catalog}.{schema}.Job j ON fj.job_id = j.job_id
  WHERE j.target_table IN (
    SELECT DISTINCT source_table 
    FROM {catalog}.{schema}.Job 
    WHERE target_table = 'gold.policy_summary'
  )
)
SELECT * FROM upstream
ORDER BY feed_name;
```

### 5.3 Error Handling

**Scenario 1: No Results**
* Response: "I couldn't find any {entity} matching your criteria. Would you like to see all {entities}?"

**Scenario 2: Ambiguous Query**
* Response: "I'm not sure which {entity} you mean. Did you mean: {option1}, {option2}?"

**Scenario 3: Access Denied**
* Response: "You don't have access to {table/schema}. Please request access from your admin."

**Scenario 4: Complex Query**
* Response: "This is a complex query. Let me break it down: {explanation}. Here's what I found: {result}"

## 6. ABC Framework Integration

### 6.1 Audit

**Audit Hook:**
```python
abc_sdk.audit(
    event="chatbot_query",
    user_id=current_user(),
    question=user_question,
    generated_sql=sql_query,
    response_time=response_duration,
    status="success" | "failed",
    result_count=row_count,
    trace_id=trace_id
)
```

**Audit Fields:**
* `event` — "chatbot_query"
* `user_id` — Databricks user ID
* `question` — natural language question
* `generated_sql` — SQL query generated by Genie
* `response_time` — duration in seconds
* `status` — success/failed
* `result_count` — number of rows returned

### 6.2 Balance

**Not applicable** — chatbot is read-only, doesn't modify data.

### 6.3 Cost Tracking

**Cost Hook:**
```python
abc_sdk.cost_track(
    operation="chatbot_query",
    user_id=current_user(),
    sql_warehouse_id=warehouse_id,
    query_duration=duration_seconds,
    rows_scanned=rows_scanned,
    bytes_scanned=bytes_scanned,
    trace_id=trace_id
)
```

**Cost Metrics:**
* SQL warehouse DBU consumption
* Query duration
* Rows/bytes scanned

## 10. Decisions Made

**Decision 1: Use Genie Space (not custom chatbot)**
* **Rationale:** Genie Space is native to Databricks, UC-integrated, requires no custom hosting
* **Alternative:** Custom LangChain/LlamaIndex chatbot (more complex, more control)
* **Trade-off:** Limited customization vs. fast time-to-value

**Decision 2: Read-only operations**
* **Rationale:** Safety first; chatbot only queries metadata, doesn't modify jobs/feeds
* **Alternative:** Allow mutations (e.g., "restart job X") — higher risk
* **Trade-off:** Safety vs. convenience

**Decision 3: Connect all metadata tables**
* **Rationale:** Users need full operational context (feeds, jobs, DQ, recon)
* **Alternative:** Connect only Feed and Job tables — limited functionality
* **Trade-off:** More surface area vs. better insights

**Decision 4: Custom system instructions**
* **Rationale:** Domain-specific language (feeds, jobs, SCD2, DQ) requires custom prompt
* **Alternative:** Generic Genie instructions — less accurate
* **Trade-off:** Maintenance overhead vs. answer quality

**Decision 5: Log all queries (ABC Audit)**
* **Rationale:** Track usage patterns, identify popular questions, audit access
* **Alternative:** No logging — less visibility
* **Trade-off:** Storage cost vs. auditability

**Decision 6: Unity Catalog access control**
* **Rationale:** Users should only see feeds/tables they have UC access to
* **Alternative:** Show all metadata regardless of UC permissions — security risk
* **Trade-off:** Complexity vs. security

**Decision 7: Response time target <5 seconds**
* **Rationale:** Interactive experience requires fast responses
* **Alternative:** Allow 10-30 second responses — poor UX
* **Trade-off:** Query complexity limits vs. user satisfaction

**Decision 8: No caching (initial version)**
* **Rationale:** Metadata changes frequently; caching adds complexity
* **Alternative:** Cache popular queries — faster but stale
* **Trade-off:** Simplicity vs. performance (revisit if needed)
